# README Generator

Generate or update the project README.md with accurate, up-to-date documentation. Follow every step in order.

## Step 1 ‚Äî Understand the project

Run these in parallel to build a full picture:
- `git log --oneline -10` ‚Äî recent commit history for context
- Read `CLAUDE.md` (if it exists) ‚Äî project overview, stack, phases
- Read `PLAN.md` or `context.md` (if they exist) ‚Äî goals and decisions
- Read `.env.example` ‚Äî environment variables and services in use
- `ls` the root ‚Äî understand top-level structure
- Read `docker-compose.yml` (or `docker-compose.prod.yml`) ‚Äî services, ports, volumes
- Read `sql/schema.sql` ‚Äî data model
- Read `public/index.php` ‚Äî routing and entry point

## Step 2 ‚Äî Explore the codebase

Run these to fill in any gaps:
- List all controllers: `ls src/controllers/`
- List all views: `find src/views -name "*.php" | sort`
- List all models: `ls src/models/`
- Check for a `Makefile`, `package.json`, or `composer.json` ‚Äî build/run commands
- Check `docker/` for any custom Dockerfiles

## Step 3 ‚Äî Draft the README

Write a README.md that includes all of the following sections (skip a section only if genuinely not applicable):

---

### `# <Project Name>`

One paragraph: what this project is, who it's for, and what problem it solves.

---

### `## Features`

Bullet list of the main capabilities. Group by role if the app has multiple user types (e.g. Buyer, Seller, Admin).

---

### `## Tech Stack`

A table or grouped list:
- **Frontend** ‚Äî e.g. HTML5, Bootstrap 5, jQuery
- **Backend** ‚Äî e.g. PHP 8, procedural + light OOP
- **Database** ‚Äî e.g. MySQL 8
- **Storage** ‚Äî e.g. Cloudflare R2
- **Payments** ‚Äî e.g. Stripe Connect
- **Infra** ‚Äî e.g. Docker, Cloudflare Tunnels, GitHub Actions

## Step 4 ‚Äî Write or update README.md

- If `README.md` does not exist: create it at the project root.
- If `README.md` already exists: read it first, then update it in place ‚Äî preserve any sections that are still accurate, update or add sections that are missing or stale.
- Use clear GitHub-flavored Markdown. No emoji unless the project already uses them.
- Keep it factual ‚Äî only document what the code actually does, not what it might do in the future.
- Do not add a "Generated by Claude" note or any AI attribution.
